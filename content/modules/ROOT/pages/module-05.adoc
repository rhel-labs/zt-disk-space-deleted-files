= Resolution

== Closing the Open Filehandle

Now that we've identified the process holding the deleted file open, we need to close that filehandle. The most straightforward way is to stop the process.

== Step 1: Record the Current Disk Usage

First, let's capture the current disk usage so we can see the difference after we fix the problem:

[source,bash,role=execute]
----
df -h / | tail -1
----

Make a mental note of the "Use%" value - we'll check this again after the fix.

== Step 2: Restart the Service

Since this is running as a systemd service, the proper way to close the open filehandle is to restart the service:

[source,bash,role=execute]
----
sudo systemctl restart business-monitor.service
----

[NOTE]
====
The `systemctl restart` command does two things:
1. **Stops** the service cleanly - this closes all open file descriptors, including the deleted log file
2. **Starts** the service fresh - it will create a new log file (if the script still tries to write to that path)

This is the professional way to handle this situation. Restarting the service ensures:
- File handles are properly closed
- The service starts in a clean state
- Any initialization code runs again
- Logs show a clear restart event for troubleshooting

In production environments, always prefer `systemctl restart` over killing processes directly when dealing with services.
====

== Step 3: Verify the Service Restarted

Confirm the service has restarted successfully:

[source,bash,role=execute]
----
systemctl status business-monitor.service
----

You should see the service is active and running, but with a new PID. Notice the service restart time is recent - this confirms the restart worked.

== Step 4: Check Disk Space Again

Now for the moment of truth - check if the disk space has been freed:

[source,bash,role=execute]
----
df -h / | tail -1
----

[TIP]
====
Compare the "Use%" value to what it was before. You should see a significant drop!

The deleted file's disk space has finally been reclaimed now that the process that had it open is gone.
====

== Step 5: Verify No Deleted Files Remain

Let's confirm there are no more deleted-but-open files (Or, at least not our super-businessey log!):

[source,bash,role=execute]
----
sudo lsof +L1 /var/log 2>/dev/null
----

You should see no output, confirming all deleted files have been cleaned up.

== Understanding What Happened

Let's recap the entire scenario:

1. **Initial State**: The business-monitor.service was running and writing to `/var/log/super-business/business-monitor.log`
2. **Scott's Action**: He deleted the log file using `rm` while the service was still running
3. **The Problem**: The file was unlinked from the directory but still open by the service, so space wasn't freed
4. **Your Investigation**: You used `df`, `systemctl`, `/proc`, and `lsof` to diagnose the issue
5. **The Fix**: You restarted the service, which closed the old filehandle and freed the space

[IMPORTANT]
====
**Best Practices to Prevent This Issue:**

1. **Never manually delete log files from running processes**
   - Use proper log rotation tools (logrotate)
   - Or stop the service first, then delete, then restart

2. **Use log rotation with signal handling**
   - Most services can receive a HUP signal to close and reopen log files
   - Example: `systemctl reload httpd` (for Apache)

3. **Monitor disk space proactively**
   - Set up alerts when disk usage exceeds thresholds
   - Don't wait for an emergency before addressing log growth

4. **Configure appropriate log retention**
   - Compress old logs
   - Delete logs after a certain age
   - Use log aggregation systems for long-term storage
====

== How to Do This Correctly

If you need to clear log space while a service is running, here's the right approach:

----
# BEST WAY: Truncate instead of delete (preserves the open file)
cat /dev/null > /var/log/service/large.log

# GOOD WAY: Restart the service
systemctl restart service

# ALSO GOOD: Stop, delete, then start
systemctl stop service
rm /var/log/service/large.log
systemctl start service

# PRODUCTION WAY: Use log rotation with signal
logrotate -f /etc/logrotate.d/service
----

[TIP]
====
**Why truncate is best**: When you truncate a file using `cat /dev/null > logfile`, the file remains at the same inode and path, but its contents are cleared. Any process with the file open continues writing to the same file, but starting from the beginning. This is instant and requires no service restart.

**Why restart works**: Restarting the service closes all old file handles and opens new ones. If the log file path is hardcoded, the service will create a new log file when it starts.
====

== Key Takeaways

✓ Deleted files can still consume disk space if a process has them open

✓ Use `lsof | grep deleted` or `lsof +L1` to find deleted-but-open files

✓ Disk space is only freed when the process closes the file or terminates

✓ Proper log rotation is essential for production systems

✓ Always investigate before deleting - understanding the root cause prevents recurrence

✓ The `/proc` filesystem is a valuable source of process information
